This page contains items that could become issues in the long term. These are either improbable problems with realistic implementations or issues that are so big a picture that we couldn't implement a practical system if we were to pay attention to all the details.

Additionally, this page contains classes that have to be extended from the current decoder implementations, and changes to the front end.

Decoder classes

  • Dictionary
    • Extended so that it can return a graph instead of an array with Pronunciations. Essential if we use dictionaries in BNF format.
    • Returns the number of alternate pronunciations.
  • Token: has to provide means for traversing the paths in both directions. The token has to know about the "children" but also about the "parents".
    • Under consideration: instead of keeping all the links, refer to the utterance graph to find out relations between tokens. In this case, we wouldn't need the link back.
  • Grammar: has to provide arcs for going in both directions.

Front end

  • Audio/cepstra files can be given as relative paths. If the relative paths are given, we also need to provide the "root" directory relative to which the audio/cepstra files are located, and, optionally, the file extension (e.g. .raw, .pcm, .sph, .cep, .mfc, .mfcc). It defaults to absolute/no extension for compatibility.
  • The front end should be able to optionally create a cepstral file and simultaneously send the cepstral vector to the trainer/recognizer. When opening a file, the front end should check if the cepstra was already produced, and if it's more recent than the audio file. If not, then it should (optionally) create the cepstra, and go ahead. In this case, maybe two sets of "root" directory/file extension should be provided.

Issues

  • NodeType - possible inconstency problem. Must be fixed.
    • BuildSentenceHMM inits SentenceHMMNode with node type and string, and these can be inconsistent. Possible solution: only use string when node type is "word".

  • The two following function are not simple if we want to have different transitions for different levels for Nphones. The update function within sentenceHMM must first find the UNIT Hmm for the state or transition, then, pass the information appropriately to the UNIT HMM. If we have multiple models or features, a state is really a compound state and can point to multiple unitHMMs. So also a transition. Then the sentencehMM must effetively update all component models.
    • SentenceHMM.updateTransition(transition, transitionstate, state.gamma);
    • SentenceHMM.updateState(state, state.gamma, vector[state.timestamp]);
