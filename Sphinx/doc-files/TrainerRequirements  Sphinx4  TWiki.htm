Trainer Requirements

This is a set of requirements for the Sphinx 4 trainer. This set of requirements is currently being constructed and has not been finalized. It's a compilation/reorganization of Bhiksha's presentation during the face-to-face meeting and Paul's recollection's from said meeting.

General requirements

  • Creates acoustic models usuable by S4
  • Allows for distributed training of models over many computers
  • Tools shall be created that will generate an external form of the acoustic model that can be used for visualization of the model. This format should be an easily parseable format such as XML.
  • Acoustic Model must optionally support GARBAGE models
  • Trainer should be capable of starting from any intermediate step: initialization, from previous models etc.

Model structure

  • Any topology of model
    • Variable number of states per model
    • Different units must be allowed to have different topologies
  • Any kind of context:
    • Context independent phonemes
    • Diphones
      • Left context diphones
      • Right context diphones
    • Triphones
      • Symmetric
      • Asymmetric
    • Tetraphones
    • Pentaphones
    • ...
  • Any given length of left and right contexts
  • Cross-word models
  • Acoustic model should allow arbitrary mixes of unit sizes (pentaphones and diphones can co-exist).

Output format

  • Models are self-describing. Models must retain information internally that describe all stages of processing that went into training the model.
  • Models may maintain enough information to allow the models to be continually updated/retrained. For example, models may not only retain the values such as means and variances, but may retain the accumulators for these values, that will allow these values to be updated in the future.
  • The acoustic model will retain enough information to ensure that only compatibile models are combined.
  • An Acoustic Model shall be stored in a single (optionally compressed) jar file
  • The acoustic model jar file may contain java class code that will provide the interface to the acoustic model. This interface includes:
    • Loading
    • Saving
    • Exporting to XML format
    • Looking up/retrieving hmms
    • Scoring hmms
  • A tool shall be created that will take the external format (XML) of an S4 acoustic model and generate an acoustic model that can be read by sphinx 2 and sphinx 3.
    • Keep in mind that sphinx 2 models have strict restrictions to the structure of models (5 state HMMs, 4 parallel input streams, semi-continuous HMMs).

Core engine

  • Must train any kind of HMM:
    • Discrete
    • Continuous with any kind of parameter sharing
      • Fully continuous, with state-level tying of entire densities
      • Sharing of means across states
      • Sharing of variances across states
      • Sharing of mixture weights across states
        • Semi-continuous and phone-dependent codebook based semicontinuous HMMs will be a special case of continuous density HMMs with tied means and variances
  • Variable state densities (Gaussians or otherwise)
    • Mixtures with different numbers of mixture components
    • Different kinds of densities at different states
  • Multiple models simultaneously:
    • Context independent
    • Diphone (L/R)
    • Triphone
    • ...
  • Simultaneous training:
    • Train multiple pronunciations simultaneously via phone nets
      • Forced-alignment needs multiple passes of training
      • Simplifies other procedures such as automatic sub-word unit identification
      • Multiple feature streams
        • Tied at any level - state, phone boundary, or word boundary
      • Multiple model types
        • e.g. phoneme and syllable models, tied at word boundary
  • Parallel training
    • Use probabilities computed from one stream to train models for a parallelly fed stream (say, use mfc-computed models / a posterioris to train plp models)
  • Passing additional info (e.g. masks, confidence in estimated feature) to trainer
  • Support different modalities of training
    • Maximum Likelihood Training
      • Baum-Welch
      • Viterbi
    • Maximum A Posteriori (MAP) training
    • Speaker adaptive training
      • Maximum Likelihood Linear Regression
    • Discriminative training
    • On-line training
      • On-line MLLR

-- PaulLamere - 19 Jan 2003
