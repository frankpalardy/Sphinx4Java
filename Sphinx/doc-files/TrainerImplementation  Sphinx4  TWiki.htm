User's Guide

The trainer can be used as a monolythic block, where the user provides data, and the system fully trains models from scratch. It can also be fine tuned by an expert user, where steps are performed according to the user's design.

A typical training sessions goes through several steps, some of them in fact iteratively. In summary, the steps would be as follows.

  • Context independent (CI) Training
    • Initialization, flat (all models are the same) or segment based (from data segmented by an existing recognizer)
    • Baum Welch (BW) accumulation
    • Normalization (norm)
    • Iterate BW/norm until log likelihood plateaus or number of iterations reached
  • Untied context dependent (CD) Training
    • Initialization, copy the base phone to each of the CD phones
    • BW accumulation
    • norm
    • Iterate BW/norm until log likelihood plateaus or number of iterations reached
  • Tied CD Training
    • Initialization, build tree to find similarities between senones, and tie them
    • BW accumulation
    • norm
    • Iterate BW/norm until log likelihood plateaus or number of iterations reached
  • Expand number of Gaussian
    • Initialization, copying the mixture densities with a small disturbance (say, means +/- 10%), so that number of mixture components doubles at each step.
    • BW accumulation
    • norm
    • Iterate BW/norm until log likelihood plateaus or number of iterations reached

As one can see, the Baum-Welch/normalization steps are the same procedure for each of the stages.

The Sphinx-4 Trainer goes through all the necessary steps to build the models. A user can indicate the initial and final stages. Therefore, one does not have to go through everything every time. For example, one could train CI models directly from another set of models, without having to go through the initialization step. The stages defined in the code are as follows (please check the API for a current list).

  • _00_INITIALIZATION Flat initialization
  • _10_CI_TRAIN CI training, running BW/norm for a number of iterations
  • _20_UNTIED_CD_TRAIN Untied CD training
  • _30_STATE_PRUNING Tree building and pruning
  • _40_TIED_CD_TRAIN Tied CD training
  • _90_CP_MODEL Copies models, possibly changing format, but does no processing

The code runs in a similar fashion to the Sphinx-4 decoder: a main method assembles all the necessary classes taking into account a properties files given at command line. A typical command line is: 

        $(JAVA_CMD) edu.cmu.sphinx.trainer.Trainer trainer.props

A typical trainer.props file is: 

############################################################
#
# Trainer properties
#
############################################################

# File containing list of audio files
edu.cmu.sphinx.trainer.ControlFile.audioFile = train.ctl

# File containing list of transcriptions, parallel to the audio file list
edu.cmu.sphinx.trainer.ControlFile.transcriptFile = train.trans

# Initial stage to be done, defaults to _00_INITIALIZATION
edu.cmu.sphinx.trainer.Trainer.initialStage = _00_INITIALIZATION

# Final stage to be done, defaults to _40_TIED_CD_TRAIN
edu.cmu.sphinx.trainer.Trainer.finalStage = _10_CI_TRAIN

# Controls the maximum number of BW iterations
edu.cmu.sphinx.trainer.Trainer.maximumIteration = 10

# Controls the minimum relative change in log likelihood to be considered as "stable"
edu.cmu.sphinx.trainer.Trainer.minimumImprovement = 0.2

############################################################
#
# Acoustic model properties
#
############################################################

# These are the same as the decoder

edu.cmu.sphinx.knowledge.acoustic.FeatureVectorLength = 39
edu.cmu.sphinx.knowledge.acoustic.format = sphinx3.ascii
edu.cmu.sphinx.knowledge.acoustic.location = file:///lab/speech/sphinx4/data/rm1_8gau_13dCep_16k_40mel_130Hz_6800Hz.ascii.zip
edu.cmu.sphinx.knowledge.acoustic.definition_file = etc/RM1_clean_13dCep_16k_40mel_130Hz_6800Hz.1800.mdef
edu.cmu.sphinx.knowledge.acoustic.data_location = cd_continuous_8gau
edu.cmu.sphinx.knowledge.acoustic.properties_file = am.props

# But these are pertinent only to the trainer. They refer to where the models will be saved

edu.cmu.sphinx.knowledge.acoustic.format.save = sphinx3.ascii
edu.cmu.sphinx.knowledge.acoustic.location.save = /lab/speech/sphinx4/data/save/

Implementation Details

This section outlines the main classes implementing the trainer. Please refer to TrainerDesign and DesignDiscussion for the pertinent discussions. In particular, refer to the Sphinx-4 blueprint for the connections between the classes and data structure. Please also refer to the Trainer Issues page for information on pending issues.

Trainer (edu.cmu.sphinx.trainer.Trainer)
The Trainer is the main class in the Trainer package. It is responsible for reading the user-defined parameters that control how the system behaves, and for calling the appropriate methods in the correct order, via the TrainerManager.

TrainerManager (edu.cmu.sphinx.trainer.TrainerManager)
It is also responsible for controlling the flow of information between the other classes. It parses (or calls a class that parses) the control file, passes the input feature file name to the Learner and the transcription and dictionary information to the UtteranceHMMBuilder. It then gets the list of states from the UtteranceHMMBUilder, gets the posterior probability back from the Learner, and passes the information to the UnitHMM for accumulation.

Learner (edu.cmu.sphinx.trainer.Learner)
It is responsible for the collection of statistics. Given a sentence HMM, it reads the incoming features (or rather, pulls the next frame(s) from the Frontend), computes the forward and/or backward probabilities, and returns the posterior probabilities that will be accumulated. In doing so, it creates and manages the forward (or alpha) variable, the backward (or beta) variable, and the normalization factors. Notice that since computation of statistics is isolated from utterance HMM creation and from the normalization, one could easily implement Baum-Welch or any other training procedure, such as Viterbi training, online training etc. It is defined as an interface, with possible implementations being Baum-Welch (requiring a forward and a backward pass), Viterbi (forward pass only) etc. This class interacts with the frontend. Some changes to the frontend will make implementation easier. This changes are:
  • Define a "file extension" property, which defaults to "none" for compatibility with current setup.
  • Define a "root" so that the paths in the transcript file (or "batch" file in the decoder) can be relative to a common branch.
  • The ability to dump cepstra to a file and at the same time send the cepstra to the trainer. This may be better accoplished by a method in the Learner that duplicates what sphinx4/tests/frontend/CepstrumFileDumper does.

FlatInitializerLearner (edu.cmu.sphinx.trainer.FlatInitializerLearner)
This class is an implementation of the Learner interface. It initializes the models with a flat distribution. It simply reads the training data, and passes the information, together with probability of 1, to the model buffers, for accumulation. Therefore, the initialization uses essentially the same methods as a normal normalization (which follows Baum-Welch).

BaumWelchLearner (edu.cmu.sphinx.trainer.BaumWelchLearner)
This class is an implementation of the Learner interface. It computes the forward and backward variables, and returns them to the TrainerManager. The accumulation of the appropriate amounts and the normalization actually are performed by the e.c.s.knowledge.acoustic.HMMPoolManager class, since the Trainer, in principle, has no knowledge of what kind of distribution the system uses.

UtteranceHMMBuilder (edu.cmu.sphinx.trainer.UtteranceHMMBuilder)
The class handles maintenance of the Utterance HMM. In the Sphinx-4 context, an "utterance" can have many "transcripts" associated with it, for example, as results of forced-aligning an input using several different streams with different models. A "transcript" contains a sequence of words for a unique stream. The UtteranceHMMBuilder contains the method that builds the HMM, and therefore handles the dictionaries, transcripts, etc. It provides the list of active states to the TrainManager.

ControlFile (edu.cmu.sphinx.trainer.ControlFile)
This interface reads the control file, parses it, and returns the appropriate information -- Utterance objects -- to the TrainManager. It maintains the information about a default dictionary (possibly defined by the user as a Sphinx-4 property, or from the control file header), the utterance-specific dictionaries, the input feature directory, and other pertinent information. It reads entries, find the dictionary used, the file name of the input feature, the transcript, the flag indicating that the file was force-aligned, and possibly a "word separation character" (for cases where the dictionary includes "words" like get ready). It then builds the full file name, sets the non-specified variables (like dictionary etc) to the default, and returns the parsed data. Possible implementations of it read the control file as described in DesignDiscussion or read separate control file and transcription in Sphinx-3's format. The choice of implementation is made at run time through Sphinx4Properties. This class, as well as the Utterance, make use of a Dictionary class. This is currently an extension of the one in e.c.s.knowledge.dictionary.

Utterance (edu.cmu.sphinx.trainer.Utterance)
It contains an utterance and all pertinent information, that is, a list of the possible transcriptions for this utterance, coming from possibly different dictionaries.

Transcript (edu.cmu.sphinx.trainer.Transcript)
It contains a single transcript for a single utterance, with information about the dictionary used, whether it is exact (i.e. if it was forced aligned or not), and the word separators.

Roadmap to CD training

From a CI trainer, the necessary steps to implement the CD trainer.

  • Initialization of models
    • Build the model definition for the CD models. The model definition associates each model with a set of state IDs. The current trainer builds this association for each CI model, when the models are initialized. This method needs to be expanded/generalized to handle CD phones. It also has to take into account that not all possible contexts appear in the training set.
    • Initialization of means, variances etc. The CI models are initialized (with the current implementation) by assigning flat distributions to transitions and the overall mean and variance to each mean and variance. In the current implementation we focus on Gaussian distributions, but this is not required. We initialize with one Gaussian per state, so mixture weight is not an issue. The initialization of CD models can take advantage of the trained CI models, copying the transition matrices and means and variances from the base phone to all its derivatives.
  • Increase in number of distribution components per state. This is done by splitting each component in two. The mixture weight is split equally, and the distribution parameters are perturbed. In the case of a Gaussian, this means a variation in the mean of each Gaussian, say to plus/minus one percent. This entails changing the means and variances pools.
  • Tree building/pruning. In the TrainerDesign, an idea is proposed about keeping the tree as part of the acoustic model, so that one can pick the convenient number of senones at will. In Sphinx-3, the tree is grown (from the untied CD models) and pruned to a specific number of senones. A direct translation from C to Java could be an initial step.
  • Creation of model definition based on the tree. Since some contexts will be collapsed together based on the tree, the model definition needs to be rebuilt.
