Transformations

  • Transformations are largely volatile
  • We do not want them to be part of the basic pool
  • Transformations will be part of an "expanded pool" object
  • The expanded pool will have storing and printing methods similar to the basic pool

Definition of Expanded Pool

  • Expanded pool object contains basic Pool and Transformations 

class ExpandedPool{
  Pool pool;
  MeanTransform[] mtxfm ;
  VarianceTransform[] vtxfm ;
}

  • Must define "with transform" objects to use ExpandedPool 

class  GaussianWithTransform extends  Gaussian{
  MeanTransform mtxfm;
  VarianceTransform vtxfm;
}

  • Similar objects must be defined to use transforms for mixture weights (MixtureGaussianWithTransform) or transition matrices (UnitHMMWithTransform). The latter two are HIGHLY unlikely to be needed ever

Accumulators must be linked to parameters

  • Each parameter type object has its own accumulator object
    • Extended parameter includes the accumulator 

Class  MeanAccumulator{
  double[] numerator;
  double denominator;
}

Class MeanWithAccumulator extends Mean {
  MeanAccumulator  meanaccum;
  public  boolean normalize()
} 


Class VarianceAccumulator{
  double[] numerator;
  double denominator;
}

Class VarianceWithAccumulator extends Variance {
VarianceAccumulator  varaccum;
public boolean normalize()
}

  • Similar extension for mixture weights, transition matrices, transforms
  • normalize() processes accumulator to compute parameter
    • Should this be in Density? E.g. variance estimation may need means. How to crosslink params for normalization?

Objects must be extended for accumulation

  • HMM and density objects in trainer are extended versions of corresponding decoder objects
    • They will contain additional methods for accumulation
  • Trainer UnitHMMs will be extended version of decoder UnitHMM
    • Additional methods for accumulation of transition matrices
    • Additional methods to pass accumulation info on to states
  • Trainer state object will be an extension of decoder state object
    • Additional methods to pass accumulation information on to densities
  • Trainer State Output Density object will extend density object
    • Additional methods to accumulate mixture weights
    • Additional methods to pass information on to component densities
  • Trainer component density object will extend basic component density object
    • Additional methods to accumulate density params
  • What is the best way to do this to minimize the amount of new code one must write to replace Gaussians with Laplacians?

Storing pools and accumulators

  • Pools may be stored along with accumulators even regularly
    • Useful for on-line learning, adaptation, etc.
  • For training, accumulators must be stored with pools
    • Simplifies code structures-Doubles accumulator data to be transferred for final normalization of models
  • Storing many copies of accumulators along with pools may cause clash in ids (pools are stored with randomized ids)
  • Need id structure resolution for this

Building Unit HMMs

  • The UnitHMMBuilder builds HMMs for a required unit based on parameter sharing information
    • Parameter sharing information must be provided externally (by the user) 

                 +--------------------+
                 |   UnitHmmBuilder   |
  Parameter      |                    |<---- Unit information
  sharing   ---->|  (Decision trees)  |
  info           |  (Means sharing)   |----> Unit HMM
                 | (Variance sharing) |
                 +--------------------+

  • UnitHMMBuilder must also build the pool
    • May be read in from a user specified file
  • Initial simple implementations
    • Decision tree based state tying, all means and variances unique (for continuous density HMMs)
    • Decision tree based state tying, common set of means and variances (for semicontinuous HMMs)
    • Decision tree based state tying, means variances common to central phone (for phone dependent codebooks in semicontinuous HMMs)

SentenceHMMBuilder

  • Shares code with decoder
  • Takes in transcript and dictionaries
    • Shouldn't dictionaries be linked with UnitHMMBuilder also?
  • Gets all required unit HMMs for a given transcript from all UnitHMMBuilders
    • Multiple UnitHMMBuilders represent multiple features or multiple models
  • Combines all unit HMMs into a sentence HMM
  • Each unitHMM refers to a single pool
  • SentenceHMM refers to all pools


  • Every unit HMM points into a pool
  • Consequently sentence HMM points into all pools

SentenceHMM is used by a "Learner"

  • Learner uses sentenceHMM and data to compute a posteriori probabilities (or equivalent terms) for all the states in the sentence HMM for each input data vector
    • These are needed to update models. Simplest implementation will use Baum-Welch
  • Some discussion about whether to have the accumulator update code in the learner, or in the unitHMMs and states.
    • The former is more logical in many ways
      • converting from Baum-Welch training to on-line training is much simpler
      • Will require change of both density object and learner to change densities (e.g Gaussian by Laplacian)
    • The latter has its own pluses and minuses
      • requires a lot more information to be passed back to the sentence HMM
      • Converting Baum-Welch to on-line training will require modification of existing density objects
      • Simplifies replacing densities, e.g. Gaussian by Laplacian
      • A point not explicitly discussed in the meeting, but that comes up: the latter also makes it simpler to train multiple models simultaneously using the struct ures we propose later in this document
    • Currently have agreed to put accumulator update on unitHMMs , states and densities.
  • But this is still open for discussion

Interaction between SentenceHMM and "Learner"

  • The learner passes the a posteriori probabilities (or equivalents) back to the sentence HMM which passes it back to unit HMMs
  • Unit HMMs update transition matrix accumulators and pass a posteriori probabilities (and data) to the states.
  • States pass a posterioris and data to the State Output Densities, which
    • update mixture weight accumulators and
    • pass a posterioris and data to Gaussians (or whatever density is used).
  • Gaussian densities update mean and variance accumulators
  • To replace Gaussians with Laplacians, one must only replace the Gaussian object with a Laplacian object that has all associated methods
  • To use on-line learning algorithms both the learner and the densities must be modified (to conform to the corresponding formulae)

Garbage Models

  • We must have the option of generating Garbage models
  • There are three options for generating Garbage models:
    • Suggestion by Dave Goddeau: Create a graph with all CI models in parallel, with a loop back from a common terminal node to a common initial node
      • Basically a uniform unigram LM structure for CI models
    • Suggestion by Rita Singh: Merge all CI models into a single "composite" CI model. Introduce a loopback from the terminating state to the initial state
    • Train an explicit garbage model
      • Some kind of Bakis topology with loopback from the end to the beginning
    • All three ideas are excellent. Only the last one needs explicit training
  • Training garbage models: Introduce them in parallel with the sentence HMM during training
    • Alternately they may be placed in parallel with every word


Training Multiple Unit Types

  • Goal: To build HMMs for multiple context dependencies simultaneously
    • e.g pentaphone, triphone , left diphone and CI models simultaneously
  • Context dependency diamond:


  • The figure can be extended for heptaphones or hexaphones of every kind etc

  • Multiple context dependent models may be trained by performing multiple passes over the data
    • One for each type of model
    • Is this better or worse? Pedro thinks this is the theoretically better option. I think perhaps not (for theoretical reasons I can explain at length )
  • Alternately, they may be trained in a single pass by training the most detailed units (with the largest contexts) and training units with smaller contexts by integrating out the contexts
  • A single pass of training may be used to train all units between the bottom and the top of the diamond
    • You can train all the units in the figure while training pentaphones
    • You can train left and right diphones along with CI phones while training triphones
  • Side note from Bent Schmidt-Nielsen. Sometimes it is useful to train "biphones" which go from the centre of one phone to the centre of the next. This is not a phoneme, but spans two phonemes.
    • More accurate than left or right diphones , though not as good as triphones
    • Much smaller than triphones, good for space/accuracy trade off

Training Multiple Units

  • This ties with the structure of allowing densities to update accumulators

  • Define basic unit HMM for the most detailed unit required (which represents the bottom of the hierarchy). Also construct unit HMMs for all the coarser units to be trained.

  • The most detailed Unit will have pointers to the HMMs for its less detailed parents in the diamond.
    • Here is an example for a triphone, left diphone and CI phone
    • Since the context dependency diagram is a DAG and not a tree, definition of this hieriarchy of units must be clearly specified 

  UnitHMM (Triphone)
     |
     +-State[]
     |
     +-TransitionMatrix
     |
     +------------UnitHMM (Left  Diphone)
     |               |
     |               +-State[]
     |               |
     |               +-TransitionMatrix
     |               |
     |               +------------UnitHMM (CI Phone)
     |               |               |
     |               |               +-State[]
     |               |               |
     |               |               +-TransitionMatrix
     |               |               |

  • The UnitHMMBuilder must build the entire chain of unit HMMs for each context dependent unit.
  • During the "Learning" phase of running the learner, the first UnitHMM in the chain provides state output density values and other scores
  • During accumulation the first UnitHMM in the chain obtains a posteriori probabilities and data from the learner and updates all of its own accumulators first
  • It then conveys the posterior probabilities and data on to the next UnitHMM on the chain
  • Constraint: all UnitHMMs must have identical number of states
  • If HMM topology is also identical they can all share transition matrices
  • Gaussian density parameters may not be shareable
    • The simplest implementation does not share density or HMM parameters across different levels of the context dependency graph (diamond).
    • It remains to be worked out whether parameter sharing across levels is actually possible or safe

Training state densities for multiple nodes in a decision tree


  • Decision trees are used to tie state output densities
    • Does not preclude sharing of parameters at finer levels too
  • We may want to train state output densities for multiple levels of nodes in the decision tree in a single pass
    • Use appropriate level of tree for decoding according to data, memory and other requirements

Training all nodes in a decision tree

  • Can be done with a hierachical structure similar to what we had for multiple units
    • The entire chain of states must be built by the UnitHMMBuilder
    • Since these represent nodes in a tree the hierarchy of state objects is very clear! 

  State for Leaf
        |
        +-StateOutputDensity
        |
        +----------State for Node
        |                |
        |                +-StateOutputDensity
                         |
                         +-State for Root
                         |       |
                         |       +-StateOutputDensity

  • The state for the leaf provides state density values to the learner
  • During accumulation the leaf state computes its accumulators and then passes the a posteriori probabilities and data to the next node in the chain
    • The next state in the chain refers to the state for a parent (or superparent) node in the tree

Generalizing

  • For any hierarchical set of units a similar structure can be used to train all of them simultaneously
    • Context dependent models through various lower contexts-Leafs through various nodes of decision tree-Trees of Gaussians or Gaussian parameters?
  • How does this interact with parameter sharing across different levels of the hierarchy?

The problem of parameter sharing

  • Sharing parameters across a hierarchy presents problems
    • e.g sharing Gaussians across different contexts (triphone and CI phone)
    • Sharing Gaussians across leaves and nodes of tree
  • During the update, each unit in a chain will update the shared parameters
    • This will result in bogus updates.
      • e.g. A triphone and its left diphone may both update a Gaussian mean for a given vector, resulting in over representation of the vector
  • Not clear how to resolve this
  • Simple solution : Do not share parameters across levels of the hierarchy
    • Triphones and diphones will never share mean/variances/mixture weights
    • Root, nodes and leaves in a decision tree will not access a common parameter
    • etc.
  • There may be a more rigorous way of sharing parameters across levels
    • e.g we know we can share transition matrices across triphones and CI phones
    • We can share means and variances across the tree in a semicontinuous HMM
  • Rita, Pedro and Bhiksha will look at how to formalize the logic for sharing, based on the math

The problem of pruning for multi-stream training

  • Like the decoder, the trainer also performs pruning to reduce computation
  • In multi-stream systems, the logic behind pruning is unclear
    • Paths that are not comparable will compete for the beam
  • This is a serious problem for multi-stream decoding too, and must be resolved in order for Phil to implement multistream decoding effectively
  • Beth will look up some earlier work on this to figure out a solution
